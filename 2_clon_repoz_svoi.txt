Работа с GitHub по SSH (клонирование + push изменений)
(Сценарий: домашняя → рабочая смена с синхронизацией через GitHub)

		клонирование + push изменений

1. Имитация рабочей ситуации
Цель: Создать локальную копию репозитория на домашнем ПК, внести изменения и синхронизировать их с рабочим 
репозиторием через GitHub.

Лучше работать в отдельной папке (например, ~/projects/), чтобы избежать конфликтов с другими файлами.
Клонировать репозиторий по HTTPS (изначально):
git clone https://github.com/Intra-git/first_project.git
⚠️ Проблема: Git пытается использовать HTTPS с паролем, хотя мы хотим работать через SSH (без пароля).
2. Настройка соединения с GitHub через SSH
Проверка текущего соединения
git remote -v
Выход:

origin  https://github.com/Intra-git/first_project.git (fetch)
origin  https://github.com/Intra-git/first_project.git (push)
→ У нас HTTPS, а нужно SSH.

Переключение на SSH
Шаг 1: Удалить текущий HTTPS-URL
git remote set-url origin git@github.com:Intra-git/first_project.git
Шаг 2: Проверка нового URL
git remote -v
Ожидаемый вывод:

origin  git@github.com:Intra-git/first_project.git (fetch)
origin  git@github.com:Intra-git/first_project.git (push)
Шаг 3: Проверка SSH-подключения
ssh -T git@github.com
Сообщение:

Hi Intra-git! You've successfully authenticated, but GitHub does not provide shell access.
✅ Соединение установлено!
(Строка X11 forwarding request failed — это нормально, она не мешает работе.)

3. Почему не работало через HTTPS
Ошибка:

remote: Invalid username or token. Password authentication is not supported for Git operations.
Причина:

GitHub отключил поддержку паролей для HTTPS-авторизации.
Нужно использовать SSH-ключи или личный токен.
4. Как теперь пушить изменения
Теперь, когда соединение настроено через SSH, просто выполняем:

git add .          # Добавить все изменения
git commit -m "Описание изменений"
git push           # Пуш без пароля
✅ Результат: Изменения успешно отправятся на GitHub!

5. Как синхронизировать изменения с рабочим репозиторием
На работе: Склонировать репозиторий с GitHub (по SSH):
git clone git@github.com:Intra-git/first_project.git
Обновить локальную копию на работе:
cd first_project
git pull origin main   # Или ваша основная ветка
Теперь в рабочем репозитории будут все ваши изменения!
📌 Шаблон для быстрой настройки SSH (если что-то забыли)
Создать и добавить SSH-ключ (если не настроен):

ssh-keygen -t ed25519 -C "ваш_электронный_адрес"
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519
Добавить ключ в GitHub:

Копируем публичный ключ который находится в фале: cat ~/.ssh/id_ed25519.pub
Вставляем в настройки GitHub: Settings → SSH and GPG keys.
Переключить репозиторий на SSH:

git remote set-url origin git@github.com:ИМЯ_ПОЛЬЗОВАТЕЛЯ/репозиторий.git
✨ Запомните ключевые моменты:
SSH = безопаснее и удобнее, чем HTTPS с паролем.
Ошибка Invalid username or token → значит, GitHub не поддерживает этот метод аутентификации.
Сообщение Hi Intra-git! → SSH настроен правильно!



			пушим на гитхаб, разные ветки

Проверяем, какие ветки есть на удалённом репозитории

git fetch origin
(Это скачает список веток с GitHub, но не произведёт merge.)

Проверяем, что теперь отображается

git branch -a
(Должно появиться remotes/origin/main.)

Устанавливаем правильную ветку по умолчанию для origin

git remote set-head origin main
(Это вместо git remote set-head origin -a.)

Сливаем изменения из удалённой main (если они есть)

git merge origin/main --allow-unrelated-histories
(Эта команда разрешает слияние, если истории репозиториев не связаны.)

Пушим изменения

git push -u origin main
⚠️ Подробнее про ошибку Недопустимая ссылка: refs/remotes/origin/main
Ошибка означает, что Git не нашёл ветку main в удалённом репозитории origin. Это может произойти, если:

На GitHub ещё нет ветки main (редко, но возможно).
У вас нет прав на чтение удалённого репозитория.
Проверьте это так:

git ls-remote --heads origin
(Если вывод не содержит main, значит, на GitHub нет этой ветки. Это может быть ошибкой на их стороне.)
